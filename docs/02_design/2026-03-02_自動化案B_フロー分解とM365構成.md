# 自動化案B: フロー分解とM365構成

作成日時: 2026-03-02 01:02:02 +09:00
作成者: Codex + GPT-5

## 1. 目的

本書は、自動化案Bである「送付元特定 + 動的帳票組版 + セクション選択型」について、実装イメージが部門横断で伝わるように、Power Automate 中心の構成と処理分解を整理したものである。

## 2. この案の位置づけ

- 作業長の追加入力を増やさず
- 送付元情報を主キーに使い
- 既存 Excel 計算ツールを活用し
- 現場向けに不要情報を落とした結果帳票を自動生成する

という要件に対して、初期費用を抑えながら自動化を高める案である。

## 3. M365 構成案

### 必須構成

- Exchange Online 共有メールボックス
- Power Automate
- SharePoint Online
- Excel Online (Business)
- Office Scripts
- 既存 Excel 計算ツール

### 条件付き構成

- Word Online (Business) または代替組版手段
- PDF 変換のための追加手段

## 4. 保管場所の分解

### SharePoint サイト

1. `受付原本` ライブラリ
   - 受信メール添付の PDF を保存する
   - 監査証跡として保持する

2. `案件マスタ` リスト
   - 送付元メールアドレス
   - 現場 ID
   - 案件 ID
   - 機種区分
   - 適用マニュアルセット
   - 適用図面セット

3. `資料マスタ` リスト
   - 共通マニュアルのセクション ID
   - 共通図面 ID
   - 適用条件
   - 版情報

4. `返信結果` ライブラリ
   - 結果帳票
   - 返信した添付一式
   - 返信ログ

5. `確認キュー` リスト
   - 案件状態
   - 要確認理由
   - 確認担当者
   - 補正値
   - 承認結果

## 5. 推奨フロー構成

### フロー1: 受付フロー

#### トリガー

- 共有メールボックスにメール受信

#### 主な処理

1. 添付有無確認
2. 添付 PDF を SharePoint へ保存
3. 送付元メールアドレスから案件候補を引く
4. 件名や送付元ルールで対象外メールを除外
5. 案件特定可否を判定
6. 確認キューへ登録

#### 補足

- 24 件 / 日程度であれば、標準的なクラウドフローで十分運用しやすい
- ここではまだ Excel を触らない

### フロー2: 計算実行フロー

#### 起動条件

- 確認キューに新規案件が入り、自動実行条件を満たした時
- または確認者が「実行可」にした時

#### 主な処理

1. 案件マスタを読み込む
2. 既存 Excel 計算ツールへ必要値を渡す
3. Office Scripts で計算実行
4. 出力セルから計算結果を取得
5. 条件に応じて資料マスタから必要資料を引く
6. 結果帳票の元データを作る

#### 実装上の注意

- Office Scripts を Power Automate で実行する場合、1 日あたりの `Run script` 呼び出し制限があるため、1 件 1 実行を原則とする
- 年 6,000 件、日 24 件であれば、単純な 1 件 1 実行なら十分低い水準である
- Excel の同時更新競合を避けるため、`1 件ずつ直列実行` または `案件ごとの作業ブック複製` を前提にする

### フロー3: 帳票組版フロー

#### 起動条件

- 計算結果が取得でき、資料候補が確定した時

#### 主な処理

1. 共通テンプレートへ案件情報、計算結果、注意事項を差し込む
2. 条件に合うマニュアルセクションを選択
3. 図面一覧や添付一覧を整形
4. 結果帳票を生成
5. 返信用ライブラリへ保存

#### 組版の実現パターン

##### パターンB-1: Premium を避ける

- 結果帳票は Excel ベースまたは事前生成済み Word/PDF を使う
- セクション選択は「どの資料を添付するか」のロジックに寄せる

##### パターンB-2: 文書生成を強化する

- Word テンプレート差し込みで案件別帳票を生成する
- 文書の見栄えは上がるが、Premium 寄りになる

### フロー4: 返信フロー

#### 起動条件

- 結果帳票生成完了

#### 主な処理

1. 返信文面を組み立てる
2. 結果帳票と必要資料を添付する
3. 返信メール送信
4. 返信ログ保存
5. 確認キューを完了へ更新

## 6. 自動化条件と停止条件

### 自動で最後まで流す条件

- 送付元から案件が一意に引ける
- 必須マスタが揃っている
- Excel 計算が正常に完了する
- 資料版が確定できる

### 停止または確認に回す条件

- 送付元で案件が複数候補になる
- 添付が複数あり対象不明
- 計算結果が閾値外または想定外
- 必要資料の版が未確定

## 7. 処理件数前提から見た実装上の考え方

### 年間 6,000 件 = 日 24 件という前提

- 1 件ごとに重い並列処理を組まなくてもよい
- 失敗率を下げるため、直列寄り設計の方がよい
- ライセンス効率は高い

### Office Scripts の扱い

- `Run script` は 1 日 24 回程度なら低負荷
- スクリプト内部で外部 API を叩かない
- 応答サイズ 5MB 制限と同期タイムアウトを意識し、入出力は最小限にする

## 8. 推奨する帳票生成戦略

### 初期

- 結果帳票は簡易でもよい
- まずは「現場で迷わないこと」を優先する
- 事前生成済み資料 + 簡易結果サマリで始める

### 次段階

- セクション単位の組み立てを導入する
- 案件条件ごとに不要情報をさらに削る

## 9. 運用責任の切り方

| 領域 | 主担当 |
| --- | --- |
| 送付元マスタ | 工事管理 |
| 資料マスタ | 技術部門 |
| Excel 計算ロジック | 技術部門 |
| フロー運用 | 情シスまたは開発運用 |
| 確認キュー運用 | 工事または技術 |
| 結果帳票レビュー | 現場代表 + 工事 |

## 10. この案のコスト感

### 標準寄り構成

- 追加コストは 0 〜 360 ドル / 年程度
- 条件: Word 系 Premium を避ける

### 文書生成強化構成

- 180 〜 1,800 ドル / 年程度
- 条件: Premium 1〜2 席または Process 1 本を見込む

## 11. この案が向いている理由

- 送付元を起点にできる今の業務と相性が良い
- 既存 Excel の価値をすぐ活かせる
- 自動化率を上げつつ、完全な AI 依存を避けられる
- 初期費用を抑えながら「現場向けに調整された帳票」を返しやすい

## 12. この案の限界

- PDF 本文抽出が必要になった時は別方式へ進化が必要
- 送付元マスタの精度が悪いと崩れる
- 文書組版を凝りすぎると、コストより運用が重くなる
