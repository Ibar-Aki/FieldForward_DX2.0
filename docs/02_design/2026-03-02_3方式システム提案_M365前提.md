# 3方式システム提案: M365前提・低イニシャルコスト重視

作成日時: 2026-03-02 01:02:02 +09:00
作成者: Codex + GPT-5

## 1. 文書の目的

本書は、要件定義を前提として、M365 利用可能、既存 Excel 計算ツールあり、初期費用は可能な限り低くしたい、という条件のもとで、システム実現方式を 3 つ提案するものである。

## 2. 共通前提

- 利用者の主入力は増やさず、送付元情報と添付資料を起点に処理する
- 既存 Excel 計算ツールは初期段階では捨てず、計算エンジンとして流用する
- 正式指示値を返すため、危険な条件では自動返信しない
- 現場に返すのは原本一式ではなく、その現場向けに絞り込んだ結果帳票と必要資料
- M365 標準コネクタでできる範囲を優先し、Premium は必要な時だけ使う

## 3. 現在の業務から見た設計の要点

- 作業長に追加入力をさせない
- 送付元情報から現場や案件を自動特定したい
- 既存 Excel の表引き、条件分岐、計算式をそのまま使いたい
- 共通マニュアルの不要部分を外した、現場向け帳票を返したい
- PDF 帳票はレイアウトゆれがある

## 4. 方式1: 送付元起点・M365標準最小構成

### 方式概要

送付元メールアドレスと現場マスタの対応を中心に据え、既存 Excel 計算ツールと事前生成済みの現場向け資料セットを使って、自動返信まで行う最小構成である。

### 想定構成

- Exchange Online 共有メールボックス
- Power Automate
- SharePoint Online
- Excel Online (Business) / Office Scripts
- 既存 Excel 計算ツール

### 処理イメージ

1. 共有メールボックスでメール受信
2. 送付元メールアドレスから現場、案件、適用マスタを特定
3. 添付 PDF は SharePoint に保管して監査証跡化
4. 既存 Excel 計算ツールへ必要値を流し込み、ブラケット個数などを計算
5. SharePoint 上の事前生成済みマニュアルセット、図面セットから対象現場向け資料を選択
6. 計算結果シートと資料セットを返信

### この方式での「現場向けに調整されたマニュアル」の作り方

- 動的に Word を組み立てるのではなく、あらかじめ現場区分ごとにマニュアル PDF セットを作成しておく
- 送付元や案件区分に応じて、返す PDF セットを選び分ける

### 利点

- 初期費用が最も低い
- M365 標準機能でまとめやすい
- 既存 Excel を最大限流用できる
- 作業長の入力ゼロに最も近い

### 欠点

- 添付 PDF の内容を本格的には使わない
- 送付元と案件の対応が崩れると誤返信リスクが高い
- マニュアルの現場最適化は事前準備負荷が高い
- 例外案件には弱い

### 向いている条件

- 送付元で案件がほぼ一意に特定できる
- 既存 Excel と案件マスタが比較的整っている
- まずは PoC を早く安く回したい

### 初期コスト感

- 3方式の中で最小
- 追加ライセンスを極力避けやすい

## 5. 方式2: 送付元起点・社内確認ハイブリッド

### 方式概要

送付元起点は維持しつつ、社内確認者が SharePoint 上の確認キューで必要項目を確認または補正してから返信する方式である。正式指示値を扱う前提では最も現実的な低コスト案である。

### 想定構成

- Exchange Online 共有メールボックス
- Power Automate
- SharePoint Online
- Excel Online (Business) / Office Scripts
- 既存 Excel 計算ツール
- SharePoint List または Microsoft Lists による確認キュー

### 処理イメージ

1. 共有メールボックスでメール受信
2. 送付元メールアドレスから案件候補を特定
3. 添付 PDF とメール情報を SharePoint に保存
4. 社内確認キューへ案件を起票
5. 確認者が必要項目を確認し、必要なら 4〜数項目だけ補正
6. 既存 Excel 計算ツールで計算実行
7. 現場向け結果帳票と資料セットを返信

### この方式での「現場向けに調整されたマニュアル」の作り方

- 方式1と同じく、低コスト重視なら事前生成済みマニュアルセットを選択する
- もしくは、共通マニュアルのセクション単位 PDF を SharePoint に持ち、案件条件で組み合わせ対象を選ぶ

### 利点

- 方式1より品質を上げやすい
- PDF の内容やレイアウトゆれに完全依存しなくてよい
- 正式指示値としての統制を組み込みやすい
- それでも Premium を避けやすい

### 欠点

- 社内確認工数が発生する
- 全自動より返信速度は落ちる
- 運用責任者を置かないと回らない

### 向いている条件

- まずは本当に使える業務導線を作りたい
- 正式指示値の品質を優先したい
- PDF 実サンプルが十分そろっていない

### 初期コスト感

- 3方式の中では低い
- 方式1より運用費は増えるが、開発とライセンスの爆発を避けやすい

## 6. 方式3: 文書AI型ハイブリッド

### 方式概要

AI Builder の document processing を使って PDF から項目抽出を自動化し、Power Automate でオーケストレーションしつつ、必要に応じて人手確認を入れる方式である。将来の項目拡張や帳票多様化には最も強い。

### 想定構成

- Exchange Online 共有メールボックス
- Power Automate
- AI Builder document processing
- 必要に応じて Document automation toolkit
- Power Apps または Dataverse ベースの確認キュー
- SharePoint Online
- Excel Online (Business) / Office Scripts
- 既存 Excel 計算ツール

### 処理イメージ

1. メール受信
2. 添付 PDF を AI Builder モデルに渡す
3. 機種、積載、停止階、昇降行程などを抽出
4. 信頼度が低いケースだけ確認キューへ送る
5. 既存 Excel 計算ツールまたは新ロジックで計算
6. 条件に合う資料を特定し、結果帳票とともに返信

### この方式での「現場向けに調整されたマニュアル」の作り方

- 将来的には文書テンプレート差し込みやセクション組み立てまで拡張しやすい
- ただし Power Automate の Word Online (Business) は Premium 扱いのため、初期費用を抑えるならここも事前生成済み資料中心にした方がよい

### 利点

- 帳票ゆれに最も対応しやすい
- 30項目へ拡張しやすい
- 自動化率を上げやすい
- 確認対象を信頼度ベースで絞り込める

### 欠点

- 3方式の中で最も初期費用が高い
- モデル学習、評価、再学習の運用が必要
- 実サンプル PDF が必要
- Dataverse や AI Builder 前提だと運用管理も重くなる

### 向いている条件

- 帳票自動抽出を本命に据えたい
- PDF サンプルを十分に集められる
- 予算と運用体制を確保できる

### 初期コスト感

- 3方式の中で最も高い
- PoC ではなく第二段階以降の拡張候補として扱うのが現実的

## 8. 推奨方針

### PoC 推奨

方式2を主推奨とする。

理由は以下の通りである。

- 正式指示値を返す前提に対して無理が少ない
- PDF 実サンプルがない現時点でも設計しやすい
- M365 標準機能と既存 Excel 活用で始めやすい
- 将来、方式3へ育てやすい

### 最小費用での先行試験

ごく小さく始めるなら方式1でよい。ただし、これは「案件特定や現場別資料返却の仮説検証」と割り切るべきであり、正式指示値の本番基盤の完成形とは見なさない方がよい。

### 将来拡張

PDF サンプルが集まり、帳票ゆれへの本格対応が必要になった段階で、方式3を検討する。

## 9. 現時点での設計上の重要判断

- 既存 Excel をすぐに捨てない
- 動的な文書生成より、まずは資料選択と結果帳票整理を優先する
- 送付元マスタ、資料版管理、確認フローを先に固める
- 方式1から方式3まで、段階的に上げられる構造にする
