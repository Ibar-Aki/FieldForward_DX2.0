# 自動化案C: AI Builder要件と精度評価計画

作成日時: 2026-03-02 01:02:02 +09:00
作成者: Codex + GPT-5

## 1. 目的

本書は、自動化案Cである「AI Builder 抽出 + 標準案件全自動停止制御型」について、必要データ、技術制約、精度評価の進め方を整理するものである。

## 2. この案の位置づけ

- 添付 PDF の中身を実際に使って自動化したい
- 送付元マスタだけに依存したくない
- 将来的に 30 項目へ広げたい

という場合の本命候補である。

## 3. 採用するモデルの考え方

### 基本方針

- 現状の帳票は「完全固定レイアウト」ではない
- 結合セルや配置ゆれがある

そのため、初期検討では `General documents` 型を基本候補とする。

### 注意

- 帳票群を分析した結果、実は 2〜3 パターンの固定帳票に収まるなら、固定テンプレート型を分けて持つ方が精度とコストは安定しやすい

## 4. Microsoft 公式要件

### 入力文書要件

- 学習用は PDF、JPG、PNG
- パスワード付き PDF は不可
- 1 文書あたり最大 20MB
- テキスト埋め込み PDF の方が精度に有利

### モデル要件

- 1 モデルあたり最大 200 collections
- Power Automate の document processing でタグ付けできるフィールドは最大 300
- ページをまたぐフィールド分割は未対応
- ページをまたぐ表の行分割は未対応

## 5. この案件で必要なデータ要件

### 最低限の対象項目

- 機種
- 積載
- 停止階
- 昇降行程

### 将来拡張を見据えた項目整理

- 現場識別項目
- 号機情報
- 機種区分
- 積載区分
- 停止階数
- 昇降行程
- 寸法関連項目
- マニュアル選定条件
- 図面選定条件

### モデル学習のための文書セット

#### 必須

- 代表的な帳票パターンごとの PDF
- 正解値付きのラベルデータ
- 送付元や案件とのひも付け結果

#### 推奨

- スキャン品質が悪い帳票
- 結合セルが多い帳票
- 例外項目のある帳票
- 同じ意味で表記ゆれがある帳票

## 6. 学習データ準備の仮定

以下は Microsoft 公式の必須数ではなく、設計上の仮定である。

| 区分 | 仮定 |
| --- | --- |
| 帳票パターン数 | まず 3〜5 パターンに分類する |
| 1パターンあたり学習用 | 15〜30 件 |
| 1パターンあたり評価用 | 5〜10 件 |
| 合計初期サンプル | 60〜150 件程度から開始 |

理由は、今回の対象が正式指示値に関わるため、少量学習で即本番というより、まず評価用データを分離して精度確認する方が重要だからである。

## 7. 精度評価の考え方

### 重要指標

| 指標 | 目的 |
| --- | --- |
| 項目別完全一致率 | 機種、積載、停止階、昇降行程ごとの抽出精度を見る |
| 案件単位の完全正答率 | 1 件の帳票がそのまま自動処理に回せるかを見る |
| 自動返信可能率 | 人手介入なしで流せる割合を見る |
| 危険な誤抽出率 | 自動返信してはいけない誤りがどれだけあるかを見る |
| 信頼度別適中率 | 信頼度閾値が妥当かを見る |

### 絶対に避けたいこと

- 誤抽出なのに自動返信へ流すこと
- 送付元と帳票内容が矛盾しているのにそのまま進むこと

## 8. 運用上の信頼度ルール仮定

以下は設計上の仮定であり、PoC で調整する。

| 条件 | 扱い |
| --- | --- |
| 全必須項目が高信頼かつ送付元マスタとも一致 | 自動返信候補 |
| 必須項目の一部が中信頼 | 確認キューへ回す |
| 必須項目の欠落、矛盾、低信頼 | 停止または手動処理 |

### 仮の閾値例

- 高信頼: 0.90 以上
- 中信頼: 0.75 以上 0.90 未満
- 低信頼: 0.75 未満

これは Microsoft の固定推奨値ではなく、今回業務向けに仮置きする運用基準である。

## 9. 精度評価の進め方

### ステップ1: 帳票分類

- 現場帳票を収集
- パターンごとに分類
- 固定帳票か、一般文書型かを判断

### ステップ2: 最小モデル作成

- 4 項目だけでモデル化
- 評価データで項目別精度を測る

### ステップ3: 送付元マスタ照合を加える

- 抽出値だけでなく、送付元との一致判定を入れる
- 自動返信可能率を再評価する

### ステップ4: 例外条件の分析

- 結合セル
- 表記ゆれ
- 複数表
- ページまたぎ

を含む失敗要因を分類する

### ステップ5: Go/No-Go 判定

- 項目精度
- 自動返信可能率
- 危険な誤抽出率
- 社内確認工数

を見て、方式2継続か、方式3進化かを判断する

## 10. 年間コスト感

### 前提

- 1 ページあたり 8 Credits
- 1 Credit = 0.01 ドル

### 年間 6,000 件あたりの概算

| ページ数仮定 | 年間処理ページ数 | AI 処理費目安 |
| --- | --- | --- |
| 3ページ | 18,000 | 1,440 ドル |
| 5ページ | 30,000 | 2,400 ドル |
| 10ページ | 60,000 | 4,800 ドル |

### 補足

- これに Power Automate Premium の座席分が加わる
- ただし利用者 500 人全員へ Premium を配る必要は通常ない
- 開発 / 本番の運用主体に絞る設計が望ましい

## 11. この案の採用条件

- 実サンプル PDF を十分に集められること
- 正解ラベルを付けられること
- 業務側が精度検証に協力できること
- 誤抽出時に自動停止させる運用を許容できること

## 12. この案を急ぎすぎない方がよい理由

- サンプル不足のまま AI を入れると、期待だけ高くなりやすい
- 正式指示値のため、抽出精度より停止条件設計の方が重要になる
- まずは方式2で業務導線を固めてからでも遅くない
