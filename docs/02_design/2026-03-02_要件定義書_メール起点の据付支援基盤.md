# 要件定義書: メール起点の据付支援基盤

作成日時: 2026-03-02 00:53:54 +09:00
作成者: Codex + GPT-5
更新日: 2026-03-04

## 1. 文書の目的

本書は、建設現場におけるエレベータ据付支援のためのメール起点共通基盤について、初期要件を整理するためのものである。

対象は、仕様票 PDF を入力として受け取り、仕様抽出、計算、資料特定、結果帳票生成、返信までを行う仕組みである。

## 2. システム目的

- 現場での手計算を減らす。
- 複数資料の探索負荷を減らす。
- 正式な据付指示値を標準ルールで返せるようにする。
- 今後、入力項目、計算ロジック、返却物を追加できる共通基盤とする。

## 3. 想定利用者

- 現場作業者
- 工事管理担当
- 設計・技術部門
- 品質・安全部門
- システム運用担当

## 4. 対象範囲

### 対象

- メールでの帳票受付
- 添付 PDF の解析
- 仕様値の抽出と正規化
- 計算ロジック実行
- 関連資料の特定
- 現場向け結果帳票の生成
- 返信メール送信
- 処理履歴の記録

### 非対象

- 現場向け常時オンライン Web UI
- CAD データの編集
- 原図作成そのもの
- 現時点での全仕様項目対応

## 5. 前提条件

- 入力帳票は PDF 形式で送付される。
- 帳票には一定の記載ルールがあるが、項目配置は固定ではない。
- Excel 由来の帳票を PDF 化したような、結合セルを含む表構造を想定する。
- 初期段階では実サンプル PDF が未提示のため、柔軟な抽出方式を前提に設計する。
- 出力値は参考値ではなく、正式な据付指示値として扱う。
- 利用者は約 500 人、年間メール送付件数は約 6,000 件、営業日 250 日換算で平均 24 件 / 日、ピーク 30 件 / 日を想定する。
- 基本運用は 1 メール 1 案件 1 PDF を前提とする。
- 同一現場から同日に複数メールが届いた場合も、それぞれ別案件として扱う。
- 送付元メールアドレスと案件の対応は、当面 `1メールアドレス = 1案件` を前提とする。
- 作業長はメール本文への自由記述を前提とせず、追加入力なしで依頼できることを前提とする。
- 既存 Excel 計算ツールは複数ファイルで構成され、入力セルと出力セルは固定されている。
- 既存 Excel 計算ツールでは VBA を使用している。
- 既存 Excel 計算ツールに外部参照や別シート参照はなく、計算結果に対する人手補正もない前提とする。
- 個人情報や機微情報は含まれない前提とする。
- 多言語対応は初期要件に含めない。

## 6. 業務フロー要件

1. 利用者が特定メールアドレスに仕様票 PDF を送る。
2. システムがメールと添付ファイルを受信する。
3. システムが帳票種別判定と項目抽出を行う。
4. システムが抽出値を内部共通キーへ正規化する。
5. システムが計算ロジックと資料特定ロジックを実行する。
6. システムが結果帳票を生成する。
7. 条件に応じて自動返信または確認フローへ回す。
8. 利用者へ結果帳票と関連資料を返信する。
9. 処理履歴、使用ルール、入力値、出力値を記録する。
10. 添付 PDF が仕様票以外と判定された場合は、エラー返信する。

## 7. 機能要件

### 7.1 メール受付機能

- 特定メールアドレスで帳票受付できること。
- 添付ファイルを取得できること。
- 送信元、件名、受信日時を記録できること。
- 対象外メールを除外できること。
- 基本は 1 メール 1 PDF を受け付けること。
- 複数 PDF 添付は初期要件では対象外とし、検知時はエラーまたは確認待ちにできること。
- 送付元メールアドレスから案件候補を引けること。

### 7.2 帳票解析機能

- PDF からテキストまたは表情報を抽出できること。
- 項目名の表記ゆれに対応できること。
- 項目配置が固定されていない帳票を扱えること。
- 結合セルを含む表構造でも必要項目を検出できること。
- 必須項目不足時にエラー判定できること。

### 7.3 仕様正規化機能

- 抽出値を内部で扱う標準キーに変換できること。
- 単位表記や表現差異を吸収できること。
- 元の抽出値と正規化後値の両方を保持できること。

### 7.4 ルール実行機能

- 条件分岐を扱えること。
- Excel 相当の表引きを扱えること。
- 計算式を実行できること。
- ロジック単位で追加、変更できること。
- 計算ルールの版を識別できること。
- 既存 Excel 計算ツールの入力セル、出力セル固定仕様を利用して自動実行できること。
- 複数 Excel 計算ツールを管理できること。

### 7.5 資料特定機能

- 共通マニュアルを条件に応じて特定できること。
- 共通図面を条件に応じて特定できること。
- 現場専用図面を条件に応じて特定できること。
- 返却対象資料の版を明示できること。

### 7.6 結果帳票生成機能

- 現場向けに整理された結果帳票を生成できること。
- 使用した仕様値、算出結果、参照資料、注意事項を帳票に出力できること。
- PDF または Excel 形式の結果帳票を生成できること。
- 初期の正本形式は PDF を前提とする。
- 添付資料の順序および命名ルールを制御できること。

### 7.7 返信機能

- 結果帳票と関連資料をメール返信できること。
- 失敗時には再送や手動対応に回せること。
- 返信ログを保持できること。
- 返信先は原則として送付元メールアドレスとする。
- 返信遅延が 3 日を超えない運用を目標とする。

### 7.8 監査・追跡機能

- どの入力からどの結果が返されたか追跡できること。
- 使用したルール版、資料版、抽出値、補正値を記録できること。
- 正式指示値としての説明責任に耐えるログを残せること。
- 最低限、使用した資料版を追跡できること。
- 添付 PDF、計算入力値、計算結果を保存できること。

## 8. 非機能要件

### 品質

- 誤抽出または誤計算の可能性がある場合は、自動返信を抑止できること。
- 重要ロジック変更時にレビュー運用を組み込めること。
- 結果返信の成功率は 99% 以上を目標とする。
- 誤返信率も 99% 以上の正確性を目標値として管理する。

### 可用性

- 利用者側はメール送受信だけで成立すること。
- 通信不安定な現場でも利用継続できること。
- 平日業務時間帯において 2 時間を超える停止を避けることを目標とする。
- 重大障害時でも翌営業日までに復旧できること。
- フォールバック時は、既存 Excel を手作業で利用し、人がメール返信できる運用を維持すること。

### 拡張性

- 新しい入力項目を追加できること。
- 新しい計算ロジックを追加できること。
- 新しい結果帳票テンプレートを追加できること。
- 新しい資料特定ルールを追加できること。
- 利用者数は大きく増えない前提だが、対象ロジック追加に耐えられる構造とすること。

### 保守性

- ルール変更箇所を特定しやすいこと。
- 運用部門が変更履歴を追えること。
- 平日日中のみ、2 名以下の運用体制でも維持できること。
- 自動再実行と手動切替の両方を設計できること。

### セキュリティ

- メール送受信における情報漏えいリスクを考慮できること。
- 対象送信元や添付制御など、最低限の受付制御ができること。
- 個人情報を前提としない設計とすること。
- M365 のみで閉じることは必須ではなく、必要に応じて社内サーバ連携を許容する。

### 保存、監査

- 添付 PDF、計算入力値、計算結果は 3 か月を基本保存期間とする。
- 月次で Excel へ転記、または月単位ファイルに集約する運用を検討できること。
- 件数、自動返信率、エラー件数、確認待ち件数、資料更新件数をレポートできること。

## 9. サンプル要件

### 9.1 サンプル対象機能

機種、積載、停止階、昇降行程からガイドレール用ブラケット個数を算出する。

### 9.2 入力項目

- 機種
- 積載
- 停止階
- 昇降行程

### 9.3 サンプル計算ロジック

- 機種係数
  - 標準 = 1
  - オーダー = 1.5
- 積載係数
  - 1トン以下 = 1
  - 1トン超え = 2
- 基本個数 = 機種係数 × 積載係数 × 停止階
- 追加個数 = 昇降行程が 10m を超える場合、10m あたり 1 個追加
- 総個数 = 基本個数 + 追加個数

### 9.4 サンプル出力

- ブラケット個数
- 入力値一覧
- 使用した計算ルール版
- 参照した資料一覧
- 注意事項

## 10. 結果帳票要件

結果帳票には最低限以下を含む。

- 現場識別情報
- 入力元帳票情報
- 抽出された仕様値
- 正規化後の仕様値
- 算出された正式指示値
- 判定に用いた条件
- 添付資料一覧
- 作成日時
- 使用ルール版
- 使用資料版
- 添付資料の順序と命名ルールに従った一覧

## 11. 例外処理要件

- 必須項目が不足している場合は自動返信しない。
- 抽出信頼度が低い場合は確認フローへ回す。
- 対応外機種や対応外条件は明示的に返答する。
- 資料版が確定できない場合は自動送信しない。
- 仕様票以外の PDF を受信した場合はエラー返信する。
- システム障害時は自動再実行を基本とし、一定回数または一定時間で解消しない場合は手動運用へ切り替える。

## 12. 運用要件

### データ更新

- 送付元メールアドレスと現場、案件の対応マスタ更新は技術部が担う。
- 既存 Excel 計算ツールの正本管理は技術部が担う。
- 共通マニュアル、共通図面、現場専用図面の更新窓口は技術部とし、元資料の責任部門が異なる場合でも技術部へ集約される運用を前提とする。
- 資料版管理はファイル単位で行う。
- 計算ロジック更新の承認者は技術部課長とする。
- 資料更新頻度は概ね 2 か月に 1 回、計算ロジックや主要資料更新は年 1〜2 回を想定する。

### レビュー

- 正式指示値に関わる計算ロジック変更はレビューを必須とする。
- 初期運用では一定条件で人手確認を挟む。
- PoC の成功判定者は技術部課長とする。
- 先行導入および本番導入の Go / No-Go は技術部部長が判断する。
- 本番前のユーザー受入テストは技術部が実施する。

### 障害対応

- 処理失敗時の再実行手段を用意する。
- 手動フォールバック手順を整備する。
- 日常問い合わせの一次窓口は技術部とする。
- 障害通知はまず社内向けとし、現場への自動障害連絡は初期要件に含めない。
- 平日日中のみの保守体制を前提とする。

## 13. 導入構成案の要件上の整理

### 構成案A: 完全自動返信

- 要件上は成立するが、初期段階では品質統制要件が厳しい。

### 構成案B: 全件レビュー

- 品質面は強いが、業務効率改善の効果が出にくい。

### 構成案C: 条件付き自動返信

- 標準条件は自動、例外は確認とする。
- 初期要件として最も現実的である。

## 14. 未確定事項

- 運用責任者を置く正式部署
- 日常運用の役割分担詳細
- 送付元情報と PDF 内容が矛盾した場合の優先順位
- M365 障害や社内ネットワーク障害時の業務継続目標
- 添付ファイル最大サイズの正確な定義
- 現場向け帳票に必ず載せる項目
- 図面やマニュアルが複数候補ある場合の優先順位
- PoC 対象現場数
- OCR 採用方針の優先順位

## 15. 初期判断

- 初期導入は少数項目のサンプル機能で成立性を検証する。
- 本番運用では、正式指示値を返す以上、完全自動化よりも追跡性と統制を優先する。
- ただし基盤自体は、将来的な自動化拡張を阻害しない構造であることを必須とする。
