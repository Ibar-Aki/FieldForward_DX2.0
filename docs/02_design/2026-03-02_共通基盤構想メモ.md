# 共通基盤構想メモ

作成日時: 2026-03-02 00:39:33 +09:00
作成者: Codex + GPT-5
更新日: 2026-03-02

## 目的

建設現場のエレベータ据付作業で使う仕様票 PDF をメールで受け取り、仕様に応じた計算、資料特定、結果帳票生成、返信までを共通的に処理できる基盤を構想する。

## この基盤で実現したいこと

- 入力手段はメールで統一する。
- 帳票 PDF のレイアウトゆれに一定範囲で耐える。
- 仕様値を抽出したうえで、案件ごとに異なる計算ロジックを適用できる。
- 必要資料を検索して添付できる。
- 結果を現場向け帳票として整形して返せる。
- 後から新しい処理ルールを追加できる。

## 共通基盤として分離すべき責務

### 1. 入力受付

- メール受信
- 添付ファイル取得
- 送信元や件名などの基本情報保持

### 2. 帳票解析

- PDF からテキストや表を抽出
- 帳票種別の判定
- 項目名ゆれや配置ゆれを吸収しながら仕様値を構造化
- 結合セルや不規則な表構造を含む、Excel 由来の PDF を扱えることを前提にする

### 3. 仕様データ正規化

- 機種、積載、停止階、昇降行程などを内部共通キーに変換
- 単位や表記ゆれを統一
- 必須項目不足時はエラーまたは確認待ち扱いにする

### 4. ルール実行

- 条件分岐
- 表引き
- 計算式評価
- 複数資料の組み合わせ判定
- Excel で実施しているロジックを段階的に移植できる表現形式を持つ

### 5. 出力生成

- 現場向け結果帳票の作成
- 関連 PDF や Excel の添付
- 判定根拠や使用した仕様値の記録

### 6. 通知と追跡

- 返信メール送信
- 処理結果ログ
- 失敗時の再実行や確認フロー

## 可変部として管理したい要素

- 抽出対象項目定義
- 入力帳票ごとの抽出ルール
- 計算ロジック
- 参照する資料セット
- 出力帳票テンプレート
- エラー時の運用フロー

## 初期サンプル機能

### サンプルテーマ

機種、積載、停止階、昇降行程から、必要なガイドレール用ブラケット個数を算出する。

### サンプル計算ルール

- 機種係数
  - 標準 = 1
  - オーダー = 1.5
- 積載係数
  - 1トン以下 = 1
  - 1トン超え = 2
- 基本個数
  - 機種係数 × 積載係数 × 停止階
- 追加個数
  - 昇降行程が 10m を超える場合、10m あたり 1 個追加
- 総個数
  - 基本個数 + 追加個数

### サンプルで検証できること

- PDF から複数項目を抽出できるか
- 抽出値を内部キーに正規化できるか
- Excel 相当の条件分岐や表引きをルールとして表現できるか
- 結果帳票を生成してメール返却できるか
- 結合セルや項目位置のばらつきがある帳票でも抽出ルールを差し替えて対応できるか

## 基本方針

- 正式な据付指示値を返す前提のため、完全自動化を急がず、入力根拠と計算根拠を追跡できる設計を優先する。
- 最初から 30 項目全てを対象にせず、少数項目で精度と運用性を検証してから拡張する。
- 業務ロジックはプログラムに埋め込まず、できる限り設定追加で拡張できる構造を目指す。
- PDF サンプル未提供の間は、固定レイアウト抽出ではなく、柔軟な項目検出前提で要件を組み立てる。

## 想定される主要リスク

- 帳票フォーマットゆれにより仕様抽出精度が下がる。
- 正式指示値として返す以上、誤抽出や誤計算の責任が大きい。
- 図面やマニュアルの版管理が不十分だと、古い資料を返す危険がある。
- Excel 依存の暗黙ロジックが多い場合、移植時に業務知識の欠落が起きやすい。

## 要件定義で先に固めるべき事項

1. 初期サンプルの入力項目一覧
2. ガイドレール用ブラケット個数算出の元ルール
3. 結果帳票に最低限載せる項目
4. 誤抽出時の扱い
5. 原資料の保管場所と版管理方法
