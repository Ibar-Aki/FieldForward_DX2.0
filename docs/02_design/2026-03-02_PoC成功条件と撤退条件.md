# PoC成功条件と撤退条件

作成日時: 2026-03-02 01:02:02 +09:00
作成者: Codex + GPT-5

## 1. 文書の目的

本メモは、メール起点の据付支援基盤について、PoC を実施する場合の成功条件、継続条件、撤退条件を事前に定義するためのものである。

## 2. 前提となる仮定

| 仮定ID | 仮定内容 | 備考 |
| --- | --- | --- |
| A-01 | 既存の Excel 計算ツールを初期の計算エンジンとして流用できる | 初期コスト抑制の前提 |
| A-02 | 送付元メールアドレスから、少なくとも現場または案件を特定できる | 完全一致でなくても、候補絞り込みに使えること |
| A-03 | 初期 PoC の対象は 1 つの計算テーマに限定する | ブラケット個数算出を想定 |
| A-04 | 正式指示値を返すため、完全自動にしない条件があってよい | 確認フローを許容する |
| A-05 | PDF 実サンプルが揃わない間は、送付元マスタ中心の仮説検証から始めてもよい | 方式1または方式2前提 |
| A-06 | SharePoint に資料とマスタを置ける | 版管理と履歴管理の前提 |
| A-07 | 現場向け帳票は最初から完璧でなくてよく、レビューで改善する | PoC では読みやすさ検証も対象 |

## 3. PoC の目的

- 送付元起点の業務導線が現場に受け入れられるか確認する
- 既存 Excel 計算ロジックを自動処理に組み込めるか確認する
- 現場向けに調整された結果帳票が業務で使えるか確認する
- 自動化と確認フローの境界を見極める

## 4. PoC 対象範囲

### 対象

- 共有メールボックスでの受付
- 添付 PDF の保存
- 送付元情報による案件特定
- 既存 Excel ツールによるブラケット個数算出
- 条件に合う資料の選定
- 結果帳票の返信

### 非対象

- 30項目の全面対応
- 全帳票レイアウト対応
- すべての機種、現場、例外条件対応
- 完全自動での本番品質保証

## 5. 成功条件

| 区分 | 成功条件 | 判定イメージ |
| --- | --- | --- |
| 業務適合 | 作業長が追加入力なしで依頼できる | 送付元と添付だけで依頼成立 |
| 技術成立 | 既存 Excel 計算ツールを自動処理へ組み込める | Office Scripts または同等手段で再利用可能 |
| 出力品質 | 返信帳票が現場で読める | 現場レビューで致命的な見づらさがない |
| 運用品質 | 例外時に人が補正できる | 確認フローで詰まらない |
| 統制 | 使用したルール、資料、入力値を追跡できる | 問い合わせ時に根拠説明できる |

## 6. 継続判断のための評価指標

| 指標 | 見る理由 | PoCでの目安 |
| --- | --- | --- |
| 送付元から案件特定できた割合 | 入力レス運用の成立性を見る | 高いほどよい |
| 自動で最後まで返せた割合 | 自動化余地を把握する | 方式1は参考値、方式2以降で重視 |
| 社内確認1件あたりの処理時間 | 運用負荷を見る | 長すぎる場合は方式見直し |
| 計算結果の差し戻し件数 | 品質リスクを見る | 0に近いことが望ましい |
| 帳票の読みやすさ評価 | 現場で使えるかを見る | 定性評価でも記録する |
| 返信所要時間 | 業務価値を見る | 現場が許容できる範囲に収まること |

## 7. 方式別の PoC 成功条件

### 方式1: 送付元起点・自動返信特化

- 送付元メールアドレスと案件マスタの対応が安定している
- 既存 Excel ツールと資料選定ロジックが自動実行できる
- 事前生成済みマニュアルセットで現場要求を満たせる

### 方式2: 送付元起点・社内確認ハイブリッド

- 確認者が少数項目を短時間で補正できる
- 確認フローを通しても現場が許容する返信速度を確保できる
- 正式指示値としての品質と運用負荷のバランスが取れる

### 方式3: 文書AI型ハイブリッド

- AI Builder モデルが対象帳票から必要項目を実用精度で抽出できる
- 信頼度の低いケースだけを確認に回せる
- AI 運用コストが想定効果に見合う

## 8. 撤退条件

| 区分 | 撤退条件 | 判断理由 |
| --- | --- | --- |
| 業務 | 送付元だけでは案件特定がほとんどできない | 入力レス運用の前提が崩れる |
| 技術 | 既存 Excel ツールを安全に自動実行できない | 初期コスト低減前提が崩れる |
| 品質 | 計算結果の誤りや差し戻しが多い | 正式指示値として危険 |
| 運用 | 社内確認負荷が高すぎる | 運用定着しない |
| データ | 資料版管理が実運用に耐えない | 誤資料送付リスクが高い |
| 導入 | 現場が返信帳票を使わない | 改善効果が出ない |

## 9. 撤退ではなく縮小継続とする条件

以下は即撤退ではなく、対象縮小で継続検討とする。

- 方式1が成立しないが、方式2なら成立する
- PDF 自動解析は不安定だが、送付元マスタと人手確認を組み合わせれば成立する
- 一部の機種や現場に限定すれば有効性がある

## 10. 推奨する判断の仕方

- 最初の Go/No-Go は「全自動化できるか」ではなく、「現場価値のある最小運用が成立するか」で判断する
- 正式指示値という性質上、誤り率よりも「危険時に止まれる設計」があるかを重視する
- PoC の成功を、そのまま全社展開の成功と見なさない

## 11. 現時点での推奨

- 最小コストで PoC を始めるなら方式1または方式2
- 正式指示値としての安全性まで考えると、PoC の主軸は方式2が最も現実的
- 方式3は PDF サンプルと予算が揃った時点で次段階として検討する
