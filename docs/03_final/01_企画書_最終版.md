## 1. エグゼクティブサマリ

建設現場のエレベータ据付作業において、作業者が複数資料を横断しながら行っている仕様確認・寸法計算・資料探索を、**メール起点で支援する共通基盤**を構築する。

作業者は仕様票 PDF を特定メールアドレスに送るだけで、仕様に応じた計算結果・必要資料・現場向け結果帳票が返信される。現場の通信環境やITリテラシに依存しない、**メールだけで完結する業務導線**を実現する。

推奨方式は **案B（半自動レビュー型）** であり、M365標準機能と既存Excel計算ツールを活用して低コストで開始し、段階的に自動化範囲を拡大する。

---

## 2. 背景と課題

### 2.1 現場の実態

- 作業者は仕様一覧表を起点に、共通マニュアル・共通図面・現場専用図面を突き合わせながら据付寸法や方法を判断している
- 現場ごとに参照資料や条件が異なり、判断に時間がかかる
- Excel による計算や担当者依存の判断が残り、作業品質のばらつきが発生している
- 仕様表帳票 PDF には一定の記載ルールがあるが、フォーマットは完全統一されていない

### 2.2 なぜメール起点か

- 現場通信環境が不安定で、Web アプリ前提の運用は現実的でない
- メールは全現場で使えており、導入障壁が最も低い
- 追加端末や専用アプリのインストールが不要
- 利用者約500人のITリテラシ差を吸収できる

### 2.3 将来への備え

- 将来は30項目程度まで条件が増える見込みがあり、個別ツール乱立では保守不能になる
- 個別機能ではなく、**計算ロジックと返却物を追加できる共通基盤**として構築する必要がある

## 4. 想定ユースケース

```
1. 作業者が現場仕様票 PDF を添付してメール送信する
2. システムが帳票の仕様値を抽出する（または送付元マスタから特定する）
3. 抽出した仕様をルールまたは計算ロジックに当てはめる
4. 必要な共通マニュアル・図面・寸法表を組み合わせる
5. 現場向けに整理された結果帳票を返信メールで返す
```

## 7. 推奨方式: 案B（半自動レビュー型）

### 7.1 推奨理由

- 正式指示値を返す前提に対して無理が少ない
- PDF 実サンプルがない現時点でも設計しやすい
- M365 標準機能と既存 Excel 活用で低コストに始められる
- 将来、案Cへ育てやすい構造にできる

### 7.2 処理の流れ

```
1. 共有メールボックスでメール受信
2. 送付元メールアドレスから案件候補を特定
3. 添付 PDF は SharePoint に保管（監査証跡化）
4. 社内確認キューへ案件を起票
5. 確認者が必要項目を確認（必要なら補正）
6. 既存 Excel 計算ツールで計算実行
7. 現場向け結果帳票と資料セットを返信
```

### 7.3 フェーズ前提

- PoC と先行導入では `送付元マスタ + 既存 Excel 計算ツール` を主軸とする
- OCR による PDF 自動抽出は拡張段階のテーマであり、PoC 成立の前提にはしない
- まずは `作業長が追加入力せずに依頼できること` と `正式指示値を安全に返せること` を優先する

### 7.4 使用するM365サービス

- Exchange Online 共有メールボックス
- Power Automate（標準コネクタ中心）
- SharePoint Online
- Excel Online (Business) / Office Scripts
- 既存 Excel 計算ツール
- SharePoint List / Microsoft Lists（確認キュー）

## 9. 導入ロードマップ

### フェーズ0: 準備

- 送付元マスタ整備
- 既存 Excel 正本確定
- 資料版管理責任者決定
- 最小帳票フォーマット決定

### フェーズ1: PoC（推奨: 案B）

- 少数項目で抽出と計算の成立性を検証
- 帳票ゆれへの耐性を確認
- 社内レビュー運用を含めた流れを構築
- **成功目標**: 作業長が追加入力なしで依頼でき、返信帳票が現場で使えること

### フェーズ2: 先行導入

- 対象現場と機種を限定して運用開始
- 結果帳票フォーマットを現場に合わせて改善
- 誤判定パターンの蓄積とルール改善
- 送付元マスタ整備率、確認待ち率、返信失敗率の定量値を採取

### フェーズ3: 拡張判定

- 実帳票サンプルが揃ったら案Cの評価開始
- AI Builder で4項目抽出を試行
- 自動停止条件の設計

### フェーズ4: 本格拡張

- AI抽出の信頼性が十分であれば案Cへ移行
- 入力項目・計算ロジックの拡張
- 返却資料種類の増加

## 10. PoC の成功条件と撤退条件

### 10.1 成功条件

| 区分 | 成功条件 |
| --- | --- |
| 業務適合 | 作業長が追加入力なしで依頼できる |
| 技術成立 | 既存 Excel 計算ツールを自動処理へ組み込める |
| 出力品質 | 返信帳票が現場で読める |
| 運用品質 | 例外時に人が補正できる |
| 統制 | 使用ルール・資料・入力値を追跡できる |

### 10.2 撤退条件

| 区分 | 撤退条件 |
| --- | --- |
| 業務 | 送付元だけでは案件特定がほとんどできない |
| 技術 | 既存 Excel ツールを安全に自動実行できない |
| 品質 | 計算結果の誤り・差し戻しが多い |
| 運用 | 社内確認負荷が高すぎる |
| データ | 資料版管理が実運用に耐えない |
| 導入 | 現場が返信帳票を使わない |

### 10.3 縮小継続とする条件

- 方式1が成立しないが方式2なら成立する
- 一部の機種や現場に限定すれば有効性がある

## 12. 主要リスクと対策

| リスク | 対策方針 |
| --- | --- |
| 帳票ゆれで抽出精度が安定しない | 初期は送付元マスタ中心。PoCで精度検証 |
| 正式指示値を返す責任の重さ | 条件付き自動返信。危険条件は確認フローへ |
| 原本資料の版管理不備 | 版不明時は自動送信しない。SharePoint版管理活用 |
| Excel ロジック移植時の暗黙知欠落 | 担当者ヒアリングとセットで進める |
| メール誤送信、情報管理 | 送信先固定、監査ログ整備 |
| PoC成功が本番成功につながらない | 成功条件と撤退条件を事前定義 |
| 技術部への業務集中 | 本番前に責任分界を明確化 |
| 後からの要求膨張 | 初期対象外を明記。追加要求は別管理 |

## 13. 批判論点への対応方針（要約）

| 想定される批判 | 対応の軸 |
| --- | --- |
| メール自動返信で正式指示値は危険 | 完全自動化ではなく条件付き。危険時は止まる設計 |
| 帳票ゆれに耐えられない | だからこそPoCを小さく始める |
| Excelの置き換えに過ぎない | 問題はExcelではなく属人化と追跡不能性 |
| 資料版管理が曖昧 | 企画が是正のきっかけになる |
| 投資効果が定量化されていない | PoCと先行導入で定量値を取る設計にする |
| 例外案件が多く効果が出ない | 標準案件の処理負荷が大きければ投資価値あり |

> **意思決定者への伝え方**: 「全部自動で返せます」と言わない。「正式指示値を返すため、危険な条件は自動化しません」と説明する。価値は単発機能ではなく、現場判断ロジックを共通資産化できる点にある。

## 14. 実行判断条件

### 14.1 着手可否の判断条件

- 既存 Excel 計算ツールの管理責任者が明確であること
- 送付元メールアドレスと案件の対応付け方針が決まっていること
- 共通マニュアル・図面の保管場所と版管理責任者が決まっていること
- 正式指示値の範囲と確認フロー範囲が合意されていること
- M365 テナント上で必要サービスの利用が可能であること

### 14.2 PoC 着手条件

- 初期対象をブラケット個数算出など少数機能に限定できていること
- 現場レビュー協力者と社内確認者が確保できること
- 失敗時の手動フォールバック手順が決められること

### 14.3 本番展開条件

- ルール変更・資料更新・障害対応の責任分界が決まっていること
- 監査ログ・資料版管理・承認記録を残せる運用が固まっていること
- 効果指標と継続改善の体制が合意されていること

## 15. 初期サンプルの位置づけ

初期 PoC では以下のサンプル計算を対象とする。

- **入力項目**: 機種、積載、停止階、昇降行程
- **出力結果**: ガイドレール用ブラケット個数
- **計算ルール**:
  - 機種係数: 標準 = 1、オーダー = 1.5
  - 積載係数: 1トン以下 = 1、1トン超え = 2
  - 基本個数 = 機種係数 × 積載係数 × 停止階
  - 昇降行程が10mを超える場合、10mあたり1個追加

このサンプルの目的は機能の完成ではなく、帳票抽出・計算ロジック化・結果帳票返信・例外処理の**成立性確認**である。
