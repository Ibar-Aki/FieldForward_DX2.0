## 1. 文書の目的

本書は、既存 Excel 計算ツールに含まれる VBA 処理のうち、PoC および先行導入で Office Scripts へ移すべき対象を整理し、何を移し、何を移さず、何を別方式へ逃がすかを明確にするための一覧である。

---

## 2. 前提

- 既存 Excel 計算ツールは複数ファイル構成である
- 入力セル、出力セルは固定である
- VBA を使用している
- 外部参照は使っていない
- 人手補正は行っていない
- 現時点では VBA ソースコードの詳細解析は未実施である

> 注意: 本書は `現行 VBA の想定機能を分類した移行方針` であり、実コードの棚卸し後に更新する前提である。

---

## 3. 移行方針

### 3.1 基本方針

- PoC では `入力値設定`、`再計算`、`出力値取得` の最小機能だけを Office Scripts 化する
- Excel の画面操作依存、イベント依存、印刷依存の VBA は移さない
- ワークフロー制御は Office Scripts ではなく Power Automate 側に寄せる
- 技術部が保守できない複雑なスクリプト構造は避ける

### 3.2 分類ルール

| 区分 | 意味 |
| --- | --- |
| A | PoC で優先的に Office Scripts へ移行する |
| B | Office Scripts ではなく Power Automate / SharePoint 側へ移す |
| C | 初期導入では移行せず、手動運用または残置する |
| D | 実コード確認後に判断する |

## 5. PoC で実装すべき最小セット

PoC で必須なのは、以下の 8 項目である。

1. `MIG-01` 機種設定
2. `MIG-02` 積載設定
3. `MIG-03` 停止階設定
4. `MIG-04` 昇降行程設定
5. `MIG-06` 再計算トリガー
6. `MIG-07` ブラケット個数読取り
7. `MIG-09` 入力値検証
8. `MIG-10` 状態初期化

---

## 6. Power Automate 側へ寄せるべき理由

以下の処理は Office Scripts に載せるより、Power Automate 側へ寄せた方が管理しやすい。

| 処理 | 理由 |
| --- | --- |
| テンプレート複製 | SharePoint のファイル運用責任と一致するため |
| エラー通知 | フロー全体の失敗文脈を持てるため |
| 承認制御 | 確認キュー状態と一体で制御できるため |
| 添付資料選定 | 資料マスタとの連携が中心になるため |
| 返信送信 | メール本文、添付順、送信先制御をまとめて扱えるため |

---

## 7. 移行しないものの扱い

初期導入で移行しない処理は、無視するのではなく、明示的に扱いを決める必要がある。

| 非移行項目 | 扱い |
| --- | --- |
| UserForm | 廃止する |
| 手動ボタン | Power Automate 起点へ置換する |
| 印刷設定 | 帳票生成側で別設計にする |
| Excel イベントマクロ | 使用しない前提へ改める |
| ActiveX / COM | 非対応として切り捨てる |

---

## 8. 実コード棚卸しで確認すべきこと

1. VBA が参照しているシート名、セル名、名前定義
2. 再計算に依存する関数、揮発性関数の有無
3. 非表示シート、保護シートの利用有無
4. イベントマクロの有無
5. 画面操作、メッセージボックス、手動確認前提処理の有無
6. Excel Online で動作しない書式 / 関数の有無

---

## 9. 受入基準

以下を満たした場合、PoC の移行は成立と判断する。

- Office Scripts でブラケット個数が既存 Excel 手計算結果と一致する
- 3 回以上連続実行しても結果ぶれがない
- 実行時間が 120 秒以内に収まる
- 実行後に次案件へ影響する状態が残らない
- Power Automate 側で失敗時停止と再実行が成立する
