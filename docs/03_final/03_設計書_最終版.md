## 1. 文書の目的

本書は、メール起点の据付支援基盤について、推奨方式である**案B（送付元特定 + 動的帳票組版 + セクション選択型）**のシステム設計を包括的に整理した設計書である。M365構成、SharePoint設計、Power Automateフロー設計、Office Scripts仕様、運用試算、PoC工数見積り、将来のOCR拡張方針を含む。

---

## 2. 案Bの設計方針

- 作業長の追加入力を増やさない
- 送付元情報を案件特定の主要キーとして使う
- 既存 Excel 計算ツールを当面の計算エンジンとして流用する
- 返却資料は原本一式ではなく、案件条件に応じた結果帳票と必要資料
- 正式指示値を返すため、危険な条件では停止または確認フローへ回す

## 5. Power Automate フロー設計

### 5.1 フロー一覧

| フロー名 | 役割 | トリガー |
| --- | --- | --- |
| F01_メール受付 | メール受信・原本保存・キュー起票 | メール受信時 |
| F02_案件特定 | 送付元から案件候補を引当 | キュー新規登録時 |
| F03_自動可否判定 | 自動実行/確認回しの振り分け | F02完了後 |
| F04_計算実行 | Excel計算ツール実行 | 自動実行待ちまたは承認後 |
| F05_帳票生成 | 結果帳票データの生成 | F04完了後 |
| F06_返信送信 | 添付付きメール返信 | F05完了後 |
| F07_失敗通知 | エラー通知・状態更新 | 例外発生時 |

### 5.2 F01_メール受付 詳細

**トリガー**: `When a new email arrives in a shared mailbox (V2)`

| 設定項目 | 推奨値 |
| --- | --- |
| Original Mailbox Address | 専用共有メールアドレス |
| Include Attachments | Yes |
| Only with Attachments | Yes |

**アクション順序**:

1. 変数初期化（受信日時、送信元、原本ID）
2. 添付有無チェック → 添付なしは終了またはエラー返信
3. 添付ファイルごとのループ → PDF以外は除外
4. SharePoint `受付原本` へファイル保存
5. メタデータ更新（原本ID、送信元、受信日時、件名、状態=受付済）
6. `確認キュー` へ新規起票

### 5.3 F02_案件特定 詳細

**トリガー**: `When an item is created` on `確認キュー`

**主要ロジック**:

1. 送信元メールで `送付元案件マスタ` を検索
2. 結果件数で分岐:
   - 1件 → 案件ID・現場ID・自動判定候補を設定
   - 0件 → 確認待ち
   - 複数件 → 確認待ち + 要確認理由を設定

### 5.4 F03_自動可否判定 詳細

**判定条件**:

- 自動返信可否 = はい
- 確認担当者が存在
- マニュアルセットID、図面セットID、ExcelテンプレートID がすべて揃っている

→ すべて揃い: `自動実行待ち` / 不足: `確認待ち`

### 5.5 F04_計算実行 詳細

**起動条件**: `キュー状態 = 自動実行待ち` または `承認結果 = 実行可`

**アクション順序**:

1. 案件マスタ読込
2. 計算テンプレートExcelの案件用複製
3. `Run script` → `setInputAndCalc`（入力セルへ値設定・再計算）
4. `Run script` → `readOutput`（出力セルから結果取得）
5. 確認キュー更新（計算結果記録、状態=返信待ち）

**実装上の注意**:

- Concurrency Control = Off を推奨
- 1日24件なら直列実行で十分
- Office Scripts `Run script` は1ユーザー1,600回/日制限 → 1件2回で余裕

### 5.6 F06_返信送信 詳細

**アクション**: `Send an email from a shared mailbox (V2)`

**返信内容に含めるもの**:

- 正式指示値であること
- 適用ルール版
- 問い合わせ先
- 結果帳票（PDF）
- 必要資料（固定順序で添付）

### 5.7 自動化条件と停止条件

| 自動で流す条件 | 停止する条件 |
| --- | --- |
| 送付元から案件が一意に引ける | 送付元で案件が複数候補 |
| 必須マスタが揃っている | 添付が複数あり対象不明 |
| Excel計算が正常完了 | 計算結果が閾値外 |
| 資料版が確定できる | 必要資料の版が未確定 |

## 6. Office Scripts 仕様

| スクリプト名 | 役割 | 備考 |
| --- | --- | --- |
| `setInputAndCalc` | 入力セルへ値を設定し再計算 | 機種・積載・停止階・昇降行程・案件ID |
| `readOutput` | 出力セルから計算結果を読み取り | ブラケット個数・計算日時・使用セル値 |
| `clearWorkbookState` | 実行後の状態初期化 | 必要に応じて |

**制約事項**:

- `Run script` の120秒同期タイムアウトを超えないこと
- 応答サイズ5MB制限
- 外部APIを叩かない
- パラメータは必要最小限にする

## 10. 将来のOCR拡張方針

### 10.1 段階的な移行計画

| 段階 | 内容 | OCR利用 |
| --- | --- | --- |
| 現在（案B） | 送付元マスタ中心。PDF は保管・監査用 | なし |
| 第2段階 | OCRmyPDF で検索可能PDF化 | 補助的 |
| 第3段階（案C） | PaddleOCR or AI Builder で項目抽出 | 本格的 |

### 10.2 OCR比較候補

| 候補 | 年間追加費用目安 | 特徴 |
| --- | --- | --- |
| AI Builder | 2,580〜2,760ドル | M365親和性が高い。PoC立ち上がりが速い |
| PaddleOCR | 0〜1,500ドル相当 | ライセンス費を抑えやすい。自前保守が必要 |
| OCRmyPDF + Tesseract | 0〜数百ドル | 検索可能PDF化には向く。項目抽出には弱い |
| NDLOCR-Lite | 0〜数百ドル | 古典籍向け。現代業務帳票には不向き |

### 10.3 OCR導入判断条件

- 実サンプルPDFを十分に集められること（60〜150件程度）
- 正解ラベルを付けられること
- 4項目（機種・積載・停止階・昇降行程）の完全一致率が実用水準であること
- 危険誤抽出率を十分に抑えられること

### 10.4 精度評価の重要指標

| 指標 | 目的 |
| --- | --- |
| 項目別完全一致率 | 各項目の抽出精度 |
| 4項目完全正答率 | 1件の自動処理可否 |
| 自動返信候補率 | 人手介入なしで流せる割合 |
| 危険誤抽出率 | 自動返信してはいけない誤りの割合 |
