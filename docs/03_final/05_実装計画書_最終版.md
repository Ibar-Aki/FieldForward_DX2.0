## 3. 実装方針

### 3.1 基本方針

- PoC では `送付元許可リスト + 確認キュー + 既存 Excel + 簡易結果帳票` で成立性を確認する
- 原則として現場作業者に追加入力を求めず、停止条件に該当した場合のみ確認担当者が確認キューで補正する
- 正式指示値を扱うため、危険条件は自動化より停止を優先する
- VBA 全体を移植せず、必要最小限だけ Office Scripts 化する
- Premium 依存は最小限に抑える

### 3.2 実装完了の定義

以下を満たした時点を、PoC 実装完了とする。

- 共有メールボックスで受信した 1 メール 1 PDF を自動受付できる
- 送付元メールアドレスで受付可否を判定し、仕様票内容と確認キュー補正で案件を確定できる
- サンプル計算対象であるブラケット個数算出を自動実行できる
- 結果帳票 PDF と必要資料を返信できる
- 停止対象ケースを 100% 確認キューへ遷移できる
- 誤返信件数が 0 件である
- PoC 対象サンプル 50 件中、正常系 45 件以上で期待どおりに処理できる
- 技術部課長および運用担当が再実行または手動切替手順を実施できる

## 9. 実装順序の判断理由

1. 送付元許可リスト運用と Excel I/O が不明確なままフローを先に作ると、後戻りが大きい
2. `F01` から `F03` を先に固めると、自動化対象と停止条件が明確になる
3. VBA 依存が最大リスクのため、Office Scripts 化は早い段階で見極める
4. 帳票体裁は後段に回し、まずは正しい結果を返せることを優先する

---

## 10. 主なリスクと対策

| リスク | 影響 | 対策 |
| --- | --- | --- |
| Excel の VBA 依存が強い | 計算自動化が止まる | 必須ロジックだけ切り出して Office Scripts 化する |
| 送付元許可リスト整備が遅れる | 受付判定と自動化率が下がる | PoC 対象者を限定し、初期対象だけ先に整備する |
| 結果帳票の見た目要求が膨らむ | 工数増大 | PoC は簡易帳票で固定し、高度化を次段階へ回す |
| 資料版が確定しない | 正式指示値として返せない | 版未確定時は停止する |
| フロー所有者が個人依存になる | 引継ぎ不能 | 共有アカウント / グループ管理で統一する |
