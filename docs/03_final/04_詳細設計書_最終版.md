## 3. 前提と重要論点

### 3.1 業務前提

- 基本は `1メール 1案件 1PDF`
- 送付元メールアドレスは許可リストで管理し、代理送信、共用アドレス、転送を許容する
- 原則として現場作業者に追加入力を求めず、必要時のみ確認担当者が確認キューで補正する
- 結果帳票の正本は PDF
- 平日日中を主運用時間帯とし、重大障害時は翌営業日までに手動フォールバックを含む業務再開を目標とする
- 問い合わせ一次窓口は技術部
- 保存対象は添付 PDF、計算入力値、計算結果

### 3.2 技術前提

- Power Automate は標準コネクタ中心
- SharePoint Online を主データストアとする
- 既存 Excel 計算ツールは複数ファイル構成
- 入力セル / 出力セルは固定
- VBA を使用している

### 3.3 重要論点

#### 論点1: VBA と Excel Online の非互換

Power Automate から Excel Online (Business) と Office Scripts を使う場合、既存 VBA をそのままクラウド実行できない可能性が高い。

#### 設計方針

- PoC / 初期導入では、計算ロジックのうち必要部分を Office Scripts で再実装する
- VBA 全体を即時移植しない
- VBA 依存が強く Office Scripts 化できない場合は、代替案として `人手フォールバック` または `社内バッチ実行` を残す

#### 論点2: 完全自動と正式指示値の両立

#### 設計方針

- 初期は標準案件のみ自動対象
- 送付元未登録、資料版未確定、計算失敗、添付異常は停止

### 3.4 フェーズ別の実装対象

| 区分 | PoC / 先行導入 | 将来拡張 |
| --- | --- | --- |
| 案件特定 | 送付元許可判定 + 確認キュー補正主軸 | PDF抽出値との突合 |
| 計算 | 既存 Excel + Office Scripts | ルールエンジン化の検討 |
| 帳票 | 簡易結果帳票 | 動的組版の高度化 |
| 判定 | 確認キュー中心 | 信頼度付き自動判定 |
| OCR | 対象外 | PaddleOCR / AI Builder 比較導入 |

## 5. コンポーネント詳細

## 5.1 受付コンポーネント

### 役割

- 共有メールボックスから対象メールを受け取る
- 添付 PDF を原本保存する
- キュー起票する

### 入力

- メールヘッダ
- 添付ファイル

### 出力

- `受付原本` ファイル
- `確認キュー` の新規行

### 主なバリデーション

- 添付有無
- PDF 拡張子
- 1 通 1 PDF

## 5.2 案件特定コンポーネント

### 役割

- 送付元メールアドレスの受付許可を判定する
- 案件特定は仕様票内容と確認キュー補正で最終確定する

### 入力

- 送信元メール
- 仕様票記載内容または確認キュー補正値

### 出力

- 案件候補ID
- 現場候補ID
- 既定値群
- 自動判定候補

### 主なバリデーション

- 許可リスト不一致 → 停止
- 重複受信判定 → 停止
- 自動返信ループ判定 → 停止

## 5.3 計算コンポーネント

### 役割

- 案件マスタ値、または将来の抽出値を Excel 計算ツールに入れる
- ブラケット個数を返す

### 入力

- 機種
- 積載
- 停止階
- 昇降行程

### 出力

- ブラケット個数
- 使用テンプレート ID
- 計算実行日時

## 5.4 資料選定コンポーネント

### 役割

- 案件条件に応じて返却資料を絞り込む

### 入力

- 案件ID
- 現場ID
- マニュアルセットID
- 図面セットID

### 出力

- 添付対象一覧

## 5.5 結果帳票生成コンポーネント

### 役割

- 計算結果と資料版をまとめた現場向け結果帳票を作る

### 初期方針

- まずは Excel から出力した結果サマリを PDF 化または簡易帳票化する
- 高度な動的組版は次段階

## 5.6 返信コンポーネント

### 役割

- 結果帳票と資料を返信する
- 送信ログを残す

## 7. 状態遷移設計

## 7.1 確認キュー状態

| 状態 | 意味 | 次状態 |
| --- | --- | --- |
| 新規 | 受付直後 | 自動実行待ち / 確認待ち / 失敗 |
| 自動実行待ち | 自動処理条件を満たす | 返信待ち / 失敗 |
| 確認待ち | 人判断待ち | 確認中 / 失敗 |
| 確認中 | 担当者が確認中 | 自動実行待ち / 返信待ち / 失敗 |
| 返信待ち | 計算と資料選定済 | 完了 / 失敗 |
| 完了 | 返信済 | - |
| 失敗 | 自動処理失敗 | 再実行対象 or 手動運用 |

## 7.2 失敗分類

| 失敗種別 | 説明 | 優先度 |
| --- | --- | --- |
| 計算失敗 | Excel / Script 実行失敗 | 最優先 |
| 返信失敗 | メール送信失敗 | 高 |
| 原本保存失敗 | 添付保存失敗 | 中 |
| 案件未特定 | マスタ不一致 | 中 |
| 資料版未確定 | 返却物を確定できない | 中 |

## 8. Power Automate 詳細設計

## 8.1 F01_メール受付

### トリガー

- `When a new email arrives in a shared mailbox (V2)`

### 主要アクション

1. 変数初期化
2. 添付有無判定
3. PDF 拡張子判定
4. `Create file` to `受付原本`
5. `Update file properties`
6. `Create item` to `確認キュー`

### 例外

- PDF 以外: エラー返信
- 添付なし: エラー返信
- 複数 PDF: 確認待ちまたはエラー
- 同一送信元 + 同一添付の短時間再送: 重複として停止
- 自動返信メールへの再返信: 再処理対象外として停止

## 8.2 F02_案件特定

### 主処理

1. `Get item` 確認キュー
2. `Get file properties` 受付原本
3. `Get items` 送付元許可リスト
4. 許可判定と重複判定で分岐

### 分岐ルール

- 許可リスト一致: 処理継続
- 許可リスト不一致: 停止
- 案件特定は仕様票記載内容を主とし、必要に応じて確認キュー補正で最終確定する

## 8.3 F03_自動可否判定

### 主処理

- 必須マスタ列存在確認
- 自動返信可否フラグ確認
- 自動実行待ち or 確認待ちへ遷移

## 8.4 F04_計算実行

### 主処理

1. Excel テンプレート複製
2. `Run script: setInputAndCalc`
3. `Run script: readOutput`
4. 結果反映

### 実装ルール

- Concurrency Control は原則オフ
- 1 件ずつ直列実行
- 実行ブックは案件単位で複製

### 再試行ルール

- 短時間で 3 回再試行
- なお失敗時は一定時間後に 1〜2 回再試行
- それでも失敗なら手動へ移行

## 8.5 F05_帳票生成

### 主処理

1. 資料マスタ検索
2. 結果データ整形
3. 結果帳票ファイル生成
4. 返信結果ライブラリ保存

### 初期方針

- 帳票体裁は最小限
- 読みやすさを優先

## 8.6 F06_返信送信

### 主処理

1. 添付対象取得
2. `Send an email from a shared mailbox (V2)`
3. 完了状態更新
4. 返信ログ記録

### 返信本文の必須要素

- 案件識別
- 正式指示値である旨
- 使用資料版
- 問い合わせ先

## 8.7 F07_失敗通知

### 主処理

- 社内通知メール送信
- キューを失敗状態へ更新

## 9. Excel / Office Scripts 詳細設計

## 9.1 テンプレート管理

| 項目 | 方針 |
| --- | --- |
| 正本 | SharePoint `計算テンプレート` に配置 |
| 実行用 | 案件ごとに複製 |
| 版管理 | ファイル単位 |
| 承認者 | 技術部課長 |

## 9.2 入出力仕様

詳細なセル番地は Excel 正本仕様書に分離するが、本設計では以下を前提とする。

| 項目 | 入出力 | 備考 |
| --- | --- | --- |
| 機種 | 入力 | 標準 / オーダー |
| 積載 | 入力 | t 単位 |
| 停止階 | 入力 | 整数 |
| 昇降行程 | 入力 | m 単位 |
| ブラケット個数 | 出力 | 正式指示値 |

## 9.3 スクリプト仕様

| スクリプト名 | 入力 | 出力 |
| --- | --- | --- |
| `setInputAndCalc` | 機種、積載、停止階、昇降行程、案件ID | 実行結果コード |
| `readOutput` | なし | ブラケット個数、計算日時 |
| `clearWorkbookState` | なし | 実行結果コード |

## 9.4 制約

- 120 秒同期タイムアウト内
- 外部 API 呼出しなし
- 応答サイズ 5MB 内

## 13. 性能、容量、体制観点の詳細設計

## 13.1 処理性能

- 日平均 24 件、ピーク 30 件 / 日

- 実測値で再調整する

## 13.3 運用体制

- 2 名以下
- 技術部中心

## 16. 結論

案B の詳細設計としては、M365 標準機能を中心に、SharePoint を主データストア、Power Automate を制御、Excel / Office Scripts を計算エンジンとする構成が現実的である。

ただし、既存 Excel が VBA を使っている点は最大の実装論点であり、PoC では必要最小限の計算ロジックを Office Scripts へ寄せて成立性を確認するのが妥当である。本番移行前には、責任分界、帳票最終仕様、矛盾時ルールを確定する必要がある。
