## 1. 使い方

- 分かる項目から記入してください
- 未定の項目は `未定`、対象外は `対象外` と記入してください
- 回答しにくいものは選択肢だけ残していただいても大丈夫です

## 2. 保守運用

### 2.1 体制

1. この仕組みの運用責任者はどの部署ですか。  
回答:未定

2. 日常運用で実際に見る人は誰ですか。`メール受付確認`、`エラー対応`、`結果確認`、`資料更新` を分けて記入してください。  
回答:資料更新は2か月に1回

3. 問い合わせ一次窓口はどこですか。現場からの問い合わせは誰が受けますか。  
回答:技術部

4. 保守体制は平日日中のみでよいですか。早朝、夜間、休日対応は必要ですか。  
回答:平日日中のみ

5. 運用引継ぎを想定していますか。想定する場合、何人程度が運用できる必要がありますか。  
回答:2名以下

### 2.2 障害対応

1. 障害時の許容停止時間はどれくらいですか。`数時間`、`当日中`、`翌営業日まで` など。  
回答:翌営業日まで

2. エラー案件の基本対応はどれですか。`自動再実行`、`担当者確認後に再実行`、`手作業へ切替`。  
回答:自動再実行、どうしても無理なら手作業

3. 再実行は何回まで許容しますか。  
回答:何回でもよい。

4. 添付保存失敗、Excel 計算失敗、返信失敗のどれが一番困りますか。優先順位をつけてください。  
回答:計算失敗が一番困る。次に返信失敗。

5. 障害時に現場へ自動連絡したいですか。それとも社内だけに通知すればよいですか。  
回答:まず社内だけ

### 2.3 マスタ、資料、ロジック管理

1. 送付元メールアドレスと現場、案件の対応マスタは誰が更新しますか。  
回答:技術部

2. 共通マニュアル、共通図面、現場専用図面の更新責任者は誰ですか。  
回答:バラバラ。技術部に連絡がくる

3. 既存 Excel 計算ツールの正本管理者は誰ですか。  
回答:技術部

4. マニュアル、図面、Excel 計算ロジックの更新頻度はどれくらいですか。`毎週`、`毎月`、`案件ごと` など。  
回答:年１～２回想定

5. 更新時の承認は必要ですか。必要なら承認者を教えてください。  
回答:技術部の課長

6. 資料版管理はどの粒度で必要ですか。`ファイル単位`、`セクション単位`、`案件セット単位` など。  
回答:ファイル単位

7. 送付元と案件の対応は `1メールアドレス = 1案件` ですか。それとも複数案件を持ちますか。  
回答:1メールアドレス、1案件

8. 自動返信してよい案件と、必ず確認に回す案件の区分はありますか。  
回答:ない

### 2.4 監査、保存、責任分界

1. どの情報を保存したいですか。`受信メール`、`添付PDF`、`計算入力値`、`計算結果`、`返信メール`、`使用資料版`、`承認履歴`。  
回答:添付PDF`、`計算入力値`、`計算結果

2. 保存期間はどれくらい必要ですか。  
回答:3か月、エクセル転記がよい。1か月①エクセルファイル

3. 計算結果が誤っていた場合の責任分界はどう考えますか。`システム運用`、`技術部門`、`現場最終確認` の考え方を教えてください。  
回答:技術部門

4. 年次監査、品質レビュー、情報システム監査などは必要ですか。必要なら頻度も教えてください。  
回答:不要

5. 運用開始後に見たいレポートは何ですか。`件数`、`自動返信率`、`エラー件数`、`確認待ち件数`、`資料更新件数` など。  
回答:全部

## 3. 非機能要件

### 3.1 性能

1. 返信時間の目標はどれくらいですか。`数分以内`、`30分以内`、`当日中` など。  
回答:当日中

2. ピーク時に同時に何件くらい来る想定ですか。  
回答:1日30件

3. 1件あたりの最大添付サイズ、最大ページ数、最大添付数の想定はありますか。  
回答:1件当たり、最大10kb、最大添付数2

4. 結果返信の失敗率や誤返信率の許容水準はありますか。  
回答:99%以上

5. 返信が遅れた場合、現場業務への影響はどの程度ですか。  
回答:3日遅れるとNG。

### 3.2 可用性、継続性

1. 可用性の目標はありますか。`平日業務時間中は止めたくない` など。  
回答:2h止めたくない。

2. M365 障害や社内ネットワーク障害時に、最低限どこまで業務継続できればよいですか。  
回答:不明

3. フォールバック時は、どの手段で業務継続したいですか。`既存Excelを手作業で使う`、`人がメール返信する` など。  
回答:`既存Excelを手作業で使い、人がメール返信する。人がマクロやバッチファイルを使用したい。

### 3.3 セキュリティ

1. 送信元制御はどこまで必要ですか。`社内ドメイン限定`、`登録済みアドレス限定`、`協力会社を含む` など。  
回答:特になし。添付されるPDFのほうがセキュリティレベルが高い。

2. 添付資料に機密区分はありますか。外部送付可否を分ける必要はありますか。  
回答:普通の情報とおなじ。

3. 個人情報や機微情報は含まれますか。含まれる場合は何ですか。  
回答:含まれない。

4. ローカルOCRサーバや社内APIを置くことに制約はありますか。  
回答:不明

5. M365 の中だけで閉じたいですか。それとも社内サーバ連携も許容できますか。  
回答:許容だが、すこしだけハードルがある。

6. 監査ログで最低限必要な項目は何ですか。`誰が送ったか`、`どの版を使ったか`、`誰が承認したか` など。  
回答:どの判をつかったか。

### 3.4 帳票、出力

1. 結果帳票の形式は `PDF固定`、`Excel固定`、`両方必要` のどれですか。  
回答:基本的にPDF

2. 結果帳票の正本はどちらですか。`PDF`、`Excel`、`メール本文`。  
回答:基本的にPDF

3. 添付資料の順序や命名にルールは必要ですか。  
回答:必要

4. 多言語対応は必要ですか。将来必要になりそうですか。  
回答:不要

### 3.5 拡張性

1. 利用者500人は今後増減しますか。件数だけ増えるのか、利用部署も増えるのかを教えてください。  
回答:増えない。

2. 将来的に帳票種類や計算ロジックが増える時、年間どのくらいの追加が想定されますか。  
回答:不明

3. 将来、据付以外の業務へ横展開する可能性はありますか。  
回答:ない。
