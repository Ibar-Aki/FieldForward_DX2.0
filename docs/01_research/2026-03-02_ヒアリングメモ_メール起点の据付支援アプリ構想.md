# ヒアリングメモ: メール起点の据付支援アプリ構想

作成日時: 2026-03-02 00:39:33 +09:00
作成者: Codex + GPT-5
更新日: 2026-03-02

## 概要

建設現場でのエレベータ据付作業において、作業者が仕様一覧表、共通エレベータ作業マニュアル、共通図面、現場専用図面を参照しながら実施している据付寸法や据付方法の判断を支援するアプリ構想を検討する。

## 現場課題の理解

- 作業者は仕様一覧表を起点に、複数の資料を突き合わせながら据付寸法や据付方法を都度計算している。
- 現場ごとに参照すべき資料や条件が異なり、作業判断に時間がかかる。
- 仕様表の帳票 PDF には一定の記載ルールがあるが、フォーマットは完全統一されていない。
- 現場の通信環境が悪く、Web アプリ前提の運用は現実的でない可能性がある。
- メールは誰でも使え、費用や運用負荷の面でも導入しやすい。

## 目指す仕組み

- 作業者が特定メールアドレス宛に仕様表付きの帳票 PDF を送る。
- システムが PDF から仕様を抽出し、現場条件に合うマニュアル、据付寸法、据付方法、関連図面の情報を特定する。
- 返信メールで、必要な PDF や Excel だけでなく、現場向けに必要箇所を整理した結果帳票も返す。
- 現場では返信メールの添付資料を見るだけで、現地での手計算や資料探索を減らせる状態を目指す。

## 想定ユースケース

1. 作業者が現場仕様票 PDF を添付してメール送信する。
2. システムが帳票の仕様値を抽出する。
3. 抽出した仕様をルールまたは計算ロジックに当てはめる。
4. 必要な共通マニュアル、共通図面、現場専用図面、寸法表を組み合わせる。
5. 返信メールで加工済み資料または検索結果を返す。

## 現時点の論点

- PDF の帳票ゆれにどこまで耐える必要があるか。
- 現場専用図面や共通図面の保管場所と検索キーをどう設計するか。
- 寸法や据付方法の導出を、単純検索で返すのか、計算して生成するのか。
- 返信物を PDF に統一するか、Excel も返すか。
- 誤判定時の責任分界と確認フローをどう置くか。

## 追加ヒアリング結果

- 返信物は原本資料の返送ではなく、現場向けに必要箇所を整理した結果帳票まで生成する想定。
- 仕様票から扱う項目は将来的に 30 項目程度まで増える想定。
- ただし初期段階では、共通基盤の成立を優先し、少数項目のサンプルユースケースから始める方針。
- この仕組みは単一の計算ツールではなく、後から「何の情報を受け取り、どのように計算し、何を返すか」を追加できる共通基盤にしたい。
- Excel で実施しているような表引き、条件分岐、計算処理を将来的に取り込める必要がある。
- 初期サンプルとしては、機種、積載、停止階、昇降行程からガイドレール用ブラケット個数を算出する程度の小さなユースケースでよい。
- 出力される値は参考値ではなく、正式な据付指示値として扱う前提。
- ブラケット個数のサンプル算出式は以下とする。
  - 機種係数: 標準 = 1、オーダー = 1.5
  - 積載係数: 1トン以下 = 1、1トン超え = 2
  - 基本式: 機種係数 × 積載係数 × 停止階
  - 追加条件: 昇降行程が 10m を超える場合、10m あたり 1 個追加
- PDF 実サンプルは現時点では未提供。帳票イメージとしては、結合セルがあり、項目配置が一定でない Excel 表を PDF 化したような資料を想定する。

## 初期アーキテクチャの方向性

- 入力チャネルはメールを主軸にする。
- メール受信後に帳票 PDF を自動解析する。
- 仕様抽出後、ルールエンジンまたは計算ロジックで必要資料を特定する。
- 出力テンプレートに当てはめ、現場向けに整形した結果帳票を生成する。
- 結果帳票と必要資料をメール返信する。
- 通信不安定現場を前提に、利用者側はメール送受信だけで成立する設計を優先する。
- 計算ロジックや出力テンプレートは後から追加可能なモジュール構造を前提にする。
- 帳票解析は、固定座標前提ではなく、見出し語、近傍セル、表構造崩れを考慮した抽出方式を前提にする。

## 確認が必要な項目

- 仕様一覧表 PDF に含まれる主要項目は何か。
- 現場専用図面はどの単位で管理されているか。
- 共通図面やマニュアルは版管理されているか。
- 返却結果に必要なのは原本資料か、現場向けに再編集された要約資料か。
- 計算結果は参考値なのか、作業指示として使う正式値なのか。
- 返信までに許容される時間はどれくらいか。
- 誰が元データを更新、保守するのか。

## 現時点で確定した事項

- 返却結果は現場向けに必要箇所を整理した結果帳票とする。
- 計算結果は正式な据付指示値として扱う。
- 初期 PoC は少数パラメータで成立する簡易サンプルから始める。
- 目標システムは、個別機能ではなく、計算ロジックと返却物を差し替え追加できる共通基盤とする。
- 初期サンプルのブラケット個数算出式は上記の仮ルールを使用して設計検討を進める。

## 次の検討ステップ案

1. 帳票 PDF から抽出すべき項目一覧を整理する。
2. 返却物の最小セットを定義する。
3. 計算ロジックと資料検索ロジックを分けて考える。
4. 誤抽出時の人手確認ポイントを設計する。
5. 要件定義書の骨子を作る。
